{props.name || 'Module Federation'} suporta Node.js out do box. Whether você são consuming módulos em runtime apenas, ou integrating em uma Webpack/Rspack pipeline de build, ele pode ser adopted com uma relativamente small amount de configuração. Este documento walks por meio de formas comuns para usar {props.name || 'Module Federation'} em Node.js.

## Visão geral

Em uma Node.js servidor aplicação, você pode carregar módulos remotos via {props.name || 'Module Federation'}. Esses módulos podem ser local arquivos criado em CommonJS format, ou remote services accessed over HTTP. Isso permite uma flexible foundation for do lado do servidor microservices, dynamic feature delivery, e shared recursos.

## Consumer

### Runtime-Only

Se você são apenas consuming módulos em Node.js e não quiser introduce bundlers like Webpack/Rspack, você pode use o runtime-apenas approach. O ideia principal é: **no plugin de build obrigatório**. Você só precisa do APIs fornecido by { props.runtime || '@module-federation/runtime' }.

Steps:

1. **Crie um MF instance** com `createInstance`.
2. **Register remotes** via o `remotes` array.
3. **Carregue módulos** via `loadRemote`.

## FAQ

### 1. What does `target: 'async-node'` do?

`target: 'async-node'` é uma Webpack alvo de build esse produz uma saída adequada para Node.js com carregamento assíncrono. Este é important for Module Federation’s modelo de carregamento dinâmico e assíncrono—especialmente quando você need top-level `await` enquanto carregamento remotes.

### 2. Why do I precisa definir `remoteType: 'script'`?

Atualmente, o MF bundler runtime apenas suporta o `script` carregamento de remote tipo, so for Node.js consumption você precisa explicitamente definir `remoteType: 'script'`.
